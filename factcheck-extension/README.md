# Разбор на фейк — расширение для браузера

Tech Vision 2026. Направление: Digital Economy & Future Tech. Зона 13: Медиа-решения (Creative Content & Media).

**Один пользователь, одна боль.** Активный пользователь соцсетей на казахском или русском языке встречает текст: пост, статью, пересылку в чате, и не может быстро понять, похоже это на фейк или манипулятивную подачу, без того чтобы гуглить и разбираться самому.

---

## Что это делает

Расширение анализирует текст или ссылку двумя независимыми способами и возвращает:

- разбор о фактической достоверности (не жёсткое «правда/ложь», а вероятностная оценка с честным «не могу проверить», если модели не хватает знаний, особенно про свежие события);
- список конкретных сомнительных утверждений, если они есть;
- признаки типичной для медиа и соцсетей манипулятивной подачи (кликбейт, анонимные источники, нагнетание паники, конспирология), найденные локально, без обращения к ИИ.

## Что это сознательно не делает

Не разбирает факты из интернета недавней давности.Это нужно потому что  у ии через open/router часто нету новой информации в БД. Это ограничение прямо прописано в системном промпте: модели явно запрещено выдумывать вердикт.

## Архитектура

```
extension/                Chrome-расширение 
  manifest.json              конфигурация, разрешения (host_permissions на бэкенд)
  background.js               подключает ввод из content.js к бэкенду(Versel) 
  content.js                   плавающая кнопка 🔍 на любой странице + панель ввода/результата
  overlay.css                  крутой вид(брутализм)

api/check.js               единственный бэкенд-файл (Vercel serverless function).
                            Вся цепочка обработки, от локальных паттернов до сборки
                            финального JSON, находится в одном месте, без сторонних оберток.
package.json                зависимости (библиотека для supabase)
```

Зависимость от Vercel неявная: папка `api/` автоматически становится serverless-функцией, отдельный `vercel.json` не нужен.

## Цепочка обработки (логика ИИ-компонентов)

Пользователь вставляет текст или ссылку. `content.js` определяет тип по регулярному выражению, `background.js` пересылает запрос на `/api/check`, а дальше внутри `api/check.js` идут четыре шага.

**1. Извлечение текста** (`extractTextFromUrl`).
Если на входе ссылка, скачивается HTML, и через `cheerio` вытаскиваются `<p>`-параграфы, которые склеиваются в текст (ограчение до 4000 символов). Если на входе уже готовый текст, он используется как есть.

**2. Локальные паттерны медиа-манипуляции** (`LOCAL_PATTERNS`, `findLocalMatches`).
Обычное сопоставление по ключевым словам, без обращения к ИИ. Быстро и предсказуемо. Пять паттернов, характерных именно для соцсетей и мессенджеров: кликбейтные заголовки («вы не поверите»), анонимные или выдуманные источники («источник в министерстве»), искусственное нагнетание паники («максимальный репост»), апелляция к размытому авторитету («учёные доказали»), конспирологическая риторика («нам не договаривают»).

**3. LLM-классификация фактической достоверности** (`buildSystemPrompt`, `classifyWithLLM`).
Модель (`openrouter/free` через OpenRouter) получает только сам текст, без локального контекста из шага 2. Фактчек и поиск манипулятивных паттернов работают как два независимых сигнала, а не смешиваются в одном промпте. Системный промпт требует:

- искать проверяемые фактические утверждения (даты, цифры, статистику, цитаты);
- честно писать «не удалось проверить достоверно», если модель не располагает данными;
- при сомнительном вердикте обязательно напоминать перепроверить через независимый источник;
- отвечать строго структурированным JSON (`riskLevel`, `riskLabel`, `signals`, `advice`).

**4. Сборка ответа** (`attachFakePatterns`).
Локальные совпадения из шага 2 добавляются в список `signals` с префиксом `[Локальный маркер]`. На защите видно, какая часть результата пришла от детерминированной проверки по ключевым словам, а какая от LLM.

Опционально каждая проверка логируется в Supabase в таблицу `checks` с текстом и результатом. 
## Формат ответа API

```json
{
  "riskLevel": "low | medium | high",
  "riskLabel": "Вероятнее всего фейк / Фактических проблем не найдено / Не удалось проверить достоверно",
  "signals": [
    "[Локальный маркер] Кликбейтный заголовок / Шок-контент: ...",
    "конкретная причина сомнения от LLM"
  ],
  "advice": "рекомендация, включая напоминание перепроверить, если riskLevel не low"
}
```

## Локальный запуск (для отладки)

```bash
npm install
npx vercel dev
```

`vercel dev` поднимает точную копию продакшен-окружения локально, включая `api/check.js` как serverless-функцию. Отдельный сервер поднимать не нужно.

Понадобится переменная окружения `OPENROUTER_API_KEY`, либо через `vercel env pull`, либо вручную в `.env.local`:
```
OPENROUTER_API_KEY=ваш_ключ
```
Опционально `SUPABASE_URL` и `SUPABASE_ANON_KEY` для логирования. Без них функция просто пропускает этот шаг.

## Деплой на Vercel

1. Закоммитьте и запушьте изменения в GitHub.
2. На [vercel.com](https://vercel.com): Project Settings → Root Directory = `factcheck-extension`.
3. Project Settings → Environment Variables: добавьте `OPENROUTER_API_KEY` (и опционально `SUPABASE_URL`, `SUPABASE_ANON_KEY`).
4. Redeploy. Vercel сам подхватит `api/check.js` без дополнительного `vercel.json`.

Проверить, что бэкенд реально работает:
```bash
curl -X POST https://ваш-проект.vercel.app/api/check \
  -H "Content-Type: application/json" \
  -d '{"type":"text","content":"Учёные доказали, что вы не поверите этому секрету!"}'
```
В ответе должен быть `signals` с `[Локальный маркер] Кликбейтный заголовок...` и `[Локальный маркер] Апелляция к абстрактному авторитету...`.

## Установка расширения (для обычных пользователей)

1. Откройте `chrome://extensions`.
2. Включите «Режим разработчика» (правый верхний угол).
3. Нажмите «Загрузить распакованное расширение» и выберите папку `extension/`(перед этим конечно же распакуйте расширение).
4. На любой странице появится плавающая кнопка 🔍 в правом нижнем углу после перезагрузки.

## Известные ограничения 

- Не работает поверх других приложений, только поверх веб-страниц. Это ограничение платформы браузерных расширений.
- Скриншоты и картинки не поддерживаются в MVP, нужна vision-модель, вынесено в roadmap.
- Извлечение текста по ссылке простое (`<p>`-теги через cheerio), не работает надёжно на сайтах с JS-рендерингом контента.
- Список `LOCAL_PATTERNS`, стартовый набор из 5 записей под медиа-манипуляции, не исчерпывающий и будет расширятся по мере CustDev.
- Модель без доступа к интернету:как я уже говорил фактчек ограничен знаниями модели на момент обучения и честно сообщает об этом ограничении в самом вердикте, а не молчит о нём.

## Roadmap (после хакатона)

- Поддержка скриншотов через vision-модель.
- Веб-поиск для фактчека в реальном времени через OpenRouter web-плагин: сейчас модель ограничена собственными знаниями.
- Расширение `LOCAL_PATTERNS` по итогам реальных CustDev-разговоров.
- Публикация в Chrome Web Store: сейчас установка только через режим разработчика.
